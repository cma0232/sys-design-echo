## Overview
Real-time collaborative editing (like Google Docs or Figma) requires solving a fundamental distributed systems problem: ensuring multiple users editing the same document concurrently end up with identical final state despite operations arriving in different orders. This topic covers two major architectural approaches—Operational Transformation (OT) and Conflict-free Replicated Data Types (CRDT)—and their trade-offs in handling concurrent edits and state convergence.

## Core Concepts

- **State Divergence Problem**: When multiple users edit concurrently, operations may arrive in different orders on different machines. Blindly applying operations in arrival order causes final state to depend on network timing rather than user intent.
- **Index-Based Operations Don't Commute**: Two users inserting at the same position produce different results depending on application order (e.g., "X hello" vs "Y hello").
- **User Intent Preservation**: The system must understand what each user intended to do and reconcile operations consistently, not arbitrarily discard edits.
- **Convergence Guarantee**: All replicas must eventually reach the identical final state regardless of operation ordering or temporary offline periods.
- **Operational Transformation (OT)**: Adjusts operations relative to previously applied operations so they can coexist while preserving intent.
- **CRDT (Conflict-free Replicated Data Type)**: Designs data structures with stable identifiers so operations naturally converge without transformation, regardless of application order.

## Architecture / Approaches

### Operational Transformation (OT)
**How it works**: 
- Introduces a central server as coordination authority
- When an operation arrives, the server transforms it relative to all previously applied operations
- Transformation adjusts indices and references to account for prior changes
- Example: If user A inserts at index 0, user B's subsequent insert at index 0 is transformed to index 1 before application

**Strengths**:
- Efficient and bandwidth-friendly in traditional client-server systems
- Successfully used in production systems like Google Docs
- Minimal metadata overhead

**Weaknesses**:
- Transformation logic becomes subtle and complex as user count increases
- Correctness guarantees are difficult to verify
- Requires central authority; difficult to support true peer-to-peer or offline-first scenarios
- Transformation logic must handle arbitrary overlapping operations

### CRDT (Conflict-free Replicated Data Type)
**How it works**:
- Models document as sequence of nodes rather than plain string indexed by position
- Each node has two pieces of information: the character and a globally unique identifier (generated by user)
- Operations reference stable node identifiers rather than shifting index positions
- When replicas sync, operations can be applied in any order; deterministic tie-breaking rules (e.g., comparing IDs) resolve conflicts
- Example: Insert operations attach new nodes after the same parent node; ID comparison determines final order

**Strengths**:
- Operations naturally commute because they reference stable identifiers, not shifting indices
- No central authority needed; supports true peer-to-peer and offline-first architectures
- Consistency guarantee is encoded into the data structure itself
- Works correctly regardless of operation arrival order

**Weaknesses**:
- Increased metadata overhead (unique identifiers per node)
- Potentially higher memory usage compared to OT
- Complexity moved from transformation logic into data structure design and metadata management

## Key Trade-offs

| Aspect | OT | CRDT |
|--------|----|----|
| **Conflict Resolution** | Centralized (server transforms operations) | Decentralized (operations naturally converge) |
| **Ordering Requirement** | Strict total order imposed by server | No ordering requirement; operations commute |
| **Architecture** | Client-server with central authority | Peer-to-peer, offline-first, local-first friendly |
| **Complexity Location** | Transformation logic (grows with user count) | Metadata and data structure design |
| **Bandwidth** | More efficient in traditional setups | May require more metadata transmission |
| **Scalability** | Harder as concurrent users increase | Better for decentralized scenarios |
| **Offline Support** | Difficult; requires server coordination | Natural support for offline edits |

## Common Interview Questions

1. **Explain the core problem that makes collaborative editing difficult. Why can't we just apply operations in arrival order?**
   - Tests understanding of state divergence and why index-based operations don't commute

2. **Walk me through how Operational Transformation resolves a conflict between two concurrent inserts at the same position.**
   - Tests ability to explain OT's transformation mechanism with concrete example

3. **What is a CRDT and how does it differ fundamentally from OT in its approach to conflict resolution?**
   - Tests understanding of the philosophical difference between centralized vs decentralized approaches

4. **In a CRDT, why do operations commute even when they arrive in different orders?**
   - Tests understanding of how stable identifiers enable commutativity

5. **Compare OT and CRDT in terms of where complexity lives in the system.**
   - Tests ability to articulate trade-off between transformation logic vs metadata overhead

6. **Why is CRDT better suited for offline-first and peer-to-peer systems than OT?**
   - Tests understanding of architectural implications of each approach

7. **What happens in an OT system as the number of concurrent users increases, and why?**
   - Tests understanding of OT's scalability challenges

8. **Design a simple CRDT for a collaborative text editor. What metadata would you store for each character?**
   - Tests ability to apply CRDT concepts to concrete design problem

## Evaluation Signals

**Strong answers demonstrate**:
- Clear understanding of why naive operation application fails (state divergence due to non-commuting operations)
- Ability to explain both OT and CRDT with concrete examples (e.g., two users inserting at same position)
- Recognition that the core challenge is preserving user intent, not just achieving any consistent state
- Understanding of architectural trade-offs: OT's centralization vs CRDT's decentralization
- Awareness of where complexity lives in each approach (transformation logic vs metadata)
- Ability to reason about which approach fits different constraints (client-server vs peer-to-peer, online vs offline-first)

**Weak answers show**:
- Confusion between the two approaches or treating them as solving different problems
- Inability to explain why operations don't commute with indices
- Missing the insight that CRDT uses stable identifiers to enable commutativity
- Treating this as a simple problem solvable with "last write wins"
- Inability to discuss trade-offs or architectural implications
- Vague explanations without concrete examples
